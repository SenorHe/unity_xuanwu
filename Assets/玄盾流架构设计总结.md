# 玄盾流架构设计总结

> 架构名称：玄盾流架构  
> 英文名称：AegisFlow Architecture  
> 适用范围：大型 Unity 商业项目、GFX / UGF 项目、仿真编辑运行系统、传统手游单体项目渐进式治理  
> 设计目标：不推倒旧系统，在可运行基础上建立边界、收敛耦合、支撑未来 3-5 年业务演进  
> 架构原则：能跑、能改、能扩

---

## 1. 架构命名解释

### 1.1 名称

```text
玄盾流架构
AegisFlow Architecture
```

### 1.2 命名含义

```text
玄：定规则
盾：守边界
流：通业务
```

### 1.3 工程解释

```text
玄 = Procedure / Domain / DC 背后的架构秩序
盾 = Service / Facade / Snapshot 建立的防腐边界
流 = UI / Command / Data / Event / Runtime 的清晰流转
```

### 1.4 一句话定义

> 玄盾流架构，是一套面向大型 Unity 项目的渐进式防腐架构。它以 Procedure 定流程，以 DomainService 守业务边界，以 DC / Repository 统一数据事实，以 Facade / Snapshot 隔离旧系统和运行态复杂度，最终让项目在不推倒重来的前提下继续能跑、能改、能扩。

## 2. 架构思想总纲

### 2.1 工程总纲

玄盾流架构不是一次“大重构方案”，而是一套大型 Unity 项目的长期演进秩序。

它解决的核心问题不是“怎么写更多功能”，而是：

```text
谁负责流程
谁负责规则
谁保存数据
谁驱动 UI
谁隔离旧系统
谁保护运行态
谁统一资源入口
谁保证系统可观测
```

最终答案是：

```text
Procedure 负责流程
ApplicationService 负责编排
DomainService 负责规则
DC / Repository 负责事实来源
Presenter 负责 UI 映射
Facade 负责旧系统防腐
Snapshot 负责运行态隔离
AssetService 负责资源入口
Observability 负责问题可见
Security 负责安全边界
```

### 2.2 玄学、佛学、道学隐喻

```text
玄以观局：看清旧系统混沌之象。
佛以破执：破除 GameMgr、单例、UI 直写、巨型类之执。
道以立序：以 Procedure、Domain、DC、Presenter、Snapshot 重建系统秩序。
```

工程翻译：

```text
用原则确定架构方向，
用边界控制复杂度，
用清晰数据流支撑业务演进。
```

## 3. 现有架构问题总结

通过分析三类 Unity 项目架构，现有系统主要问题集中在以下方面。

### 3.1 职责膨胀

典型表现：

```text
GameMgr 变成全局中枢
Procedure 承担业务逻辑和协议拼装
EntityRepository 承担 Repository + DomainService + EventBroadcaster
MrCSave 成为巨型协议分发器
GenerateEntityFactory 同时负责创建、恢复、分组、转换
```

### 3.2 数据流混乱

典型表现：

```text
UI 直接修改 Dao / PlayerState / Repository
运行态直接读取编辑态数据
网络回包直接驱动 UI
保存过程依赖当前全局模型状态
```

### 3.3 全局单例过重

典型表现：

```text
GameMgr.inst 到处访问
NLDataManager / MrCDataManager 直接暴露全局 Save
GameEntry.Instance 存在 MonoBehaviour new 风险
```

### 3.4 异步语义不清

典型表现：

```text
async void OnUpdate
ProcessEvent 返回 Task 但未真正 await
实体加载未明确 await
Task.WhenAll 看似等待行为完成，实际只是派发完成
```

### 3.5 资源入口分散

典型表现：

```text
Resources.Load 到处散落
ResourceMgr / AssetsDLMgr / UGF Resource 混用
Lua / 热更 / 内置资源边界不清
资源路径字符串缺少集中治理
```

### 3.6 UI 与业务耦合

典型表现：

```text
UI 直接访问 PlayerState
UI 直接写 Repository
UI 直接拼网络协议
UI 事件订阅生命周期靠人工维护
字符串窗口名和反射调用缺少编译期约束
```

## 4. 可复用优势设计

| 来源 | 可复用设计 | 新架构采纳方式 |
|---|---|---|
| GFX / UGF 项目 | Procedure FSM | 保留为统一流程骨架 |
| GFX / UGF 项目 | Event / UI / Resource / Entity 组件能力 | 作为底层基础设施 |
| Dao / DC 模式 | 数据中心作为事实来源 | 升级为分域 DC |
| EntityRepository | 实体集中管理 | 保留数据容器职责，剥离业务规则 |
| EventProcessor + Handler | 事件分发模式 | 强类型化并明确 async 语义 |
| DataSaver | 原子保存策略 | 纳入统一 SaveSystem |
| UIWindowMgr | UI 栈管理思想 | 强类型化 UIFormBase + Presenter |
| PlayerState | 玩家状态集中管理 | 渐进迁移为 PlayerDC / DomainService |
| BattleField + Lua | 战斗核心相对独立 | 保留战斗内核，治理输入输出边界 |
| ResourceFacade 思路 | 资源入口收敛 | 新增 AssetService 统一资源访问 |

## 6. 核心分层职责

### 6.1 Bootstrap Layer

核心模块：

```text
GameEntry
AppBootstrap
EnvironmentService
GlobalExceptionHandler
LogService
AppLifecycleService
```

职责：

```text
只负责启动和初始化
不承载业务逻辑
不直接操作 UI / DC / Repository
```

---

### 6.2 Procedure FSM Layer

推荐流程：

```text
ProcedureLaunch
  ↓
ProcedureSplash
  ↓
ProcedurePreload
  ↓
ProcedureLogin
  ↓
ProcedureHub
  ├── ProcedureSimulationEditor
  ├── ProcedureSimulationRuntime
  ├── ProcedureGameScene
  ├── ProcedureBattle
  └── ProcedureChatRoom
```

Procedure 允许：

```text
打开 / 关闭必要 UI
启动 ApplicationService
监听关键流程事件
切换 Procedure
```

Procedure 禁止：

```text
直接拼协议
直接写 DC / Repository
直接处理复杂业务规则
直接维护大量 UI 状态
```

---

### 6.3 Application Service Layer

核心服务：

```text
LoginAppService
SceneAppService
SimulationAppService
BattleAppService
SaveAppService
AssetAppService
ChatAppService
```

职责：

```text
组织业务用例
协调 DomainService / DC / Network / UI / Runtime
不保存核心状态
不直接操作具体 UI 控件
```

---

### 6.4 Domain Anti-Corruption Layer

核心服务：

```text
ModelDomainService
EntityDomainService
AgentDomainService
RuntimeSnapshotService
PlayerDomainService
RoomDomainService
BattlePrepareDomainService
TableDomainService
PermissionDomainService
```

职责：

```text
处理业务规则
屏蔽旧系统细节
阻止 UI / Procedure 直接访问底层数据
为旧 GameMgr / PlayerState / Repository 建立防腐边界
```

---

### 6.5 Data Center / Repository Layer

推荐数据中心：

```text
AccountDC
PlayerDC
RoomDC
ChatDC
SimulationDC
RuntimeDC
BattleDC
AssetDC
SettingDC
```

推荐 Repository：

```text
EntityRepository
ModelRepository
RuntimeRepository
ConfigRepository
```

原则：

```text
DC 是单一事实来源
Repository 只做数据容器
业务规则放 DomainService
UI 刷新由 Presenter 监听数据变化
```

---

### 6.6 Communication Layer

通信层分三类：

```text
Network
Event
Command
```

网络链路：

```text
ApplicationService
  ↓
ReqService
  ↓
NetworkClient
  ↓
Server
  ↓
DataProcessorRegistry
  ↓
ConcreteDataProcessor
  ↓
DC.Save()
```

事件边界：

| 场景 | 推荐方式 |
|---|---|
| 框架加载、场景加载、UI 打开 | GF EventArgs |
| 跨领域业务通知 | DomainEvent |
| UI 内部交互 | UI Local Event |
| 高频实体同步 | RuntimeCommand |
| 网络回包 | DataProcessor |
| 仿真事件 | SimulationEventProcessor + Handler |

---

### 6.7 Presentation Layer

核心模块：

```text
UIFormBase
Presenter
ViewModel
ViewData
SUI Component
RTE Adapter
UICommand
```

UI 原则：

```text
UI 只展示 ViewData
UI 只发 Command
UI 不直接写 DC
UI 不直接访问 Repository
UI 不直接拼协议
UI 不直接修改 PlayerState
```

推荐流向：

```text
DC / Repository
  ↓
Presenter
  ↓
ViewData
  ↓
UIForm / SUI
```

用户操作流：

```text
UI Button
  ↓
UICommand
  ↓
ApplicationService
  ↓
DomainService
  ↓
DC / Repository
```

---

### 6.8 Runtime Execution Layer

仿真运行态：

```text
SimulationRuntimeController
RuntimeSnapshot
SimulationEntityBootstrapper
SimulationStepScheduler
SimulationPlaybackState
SimulationEventProcessor
GF Entity FSM
```

战斗运行态：

```text
BattleInputBuilder
BattleField
BattleScriptRuntime / Lua
BattleEventBus
BattleResultCollector
BattleReplayRecorder
```

核心原则：

```text
运行态不直接读编辑态 Repository
运行态使用 RuntimeSnapshot
战斗态使用 BattleInput
运行态执行结果写 RuntimeDC
编辑态数据不被运行态污染
```

---

### 6.9 Resource Layer

核心模块：

```text

ResourceFacade
ManifestService
LuaAssetService
```

```text
所有资源加载统一走 AssetService
```

## 9. 新旧架构迁移方案

### 9.1 迁移原则

```text
不推倒重来
旧系统稳定运行
新功能进入新架构
旧模块通过 Adapter 接入
高风险模块最后迁移
```

---

### 9.2 第一阶段：止血期

目标：

```text
阻止耦合继续扩大
```

动作：

```text
修复 GameEntry MonoBehaviour 单例问题
新增 DomainService / ResourceFacade / UIFormBase
新 UI 禁止直接访问 GameMgr / PlayerState / Repository
新资源加载统一走 AssetService
事件订阅必须 OnEnable / OnDisable 配对
MrCData / NLData 监听去重
```

---

### 9.3 第二阶段：边界建设期

目标：

```text
建立新旧系统防腐层
```

动作：

```text
为 GameMgr 建立 LegacyGameAdapter
为 PlayerState 建立 PlayerDomainService
为 EntityRepository 建立 EntityDomainService
为 MrCSave 建立 DataProcessorRegistry
为 GenerateEntityFactory 拆出 Resolver / Binder / RestoreService
为 Resources / ResourceMgr 建立 ResourceFacade
```

---

### 9.4 第三阶段：核心流程治理期

目标：

```text
拆最重、风险最高的流程
```

动作：

```text
拆 ProcedureLogin
拆 ProcedureSimulationRunTime
拆 MrCSave.DoProcessData
SceneType 路由表化
保存恢复统一走 SaveAppService
```

---

### 9.5 第四阶段：资源与运行态升级期

目标：

```text
提升性能和稳定性
```

动作：

```text
新资源统一走 Addressables / UGF Resource Adapter
旧 Resources.Load 逐步替换
RuntimeSnapshot 隔离编辑态 / 运行态
仿真事件明确 Dispatch / Execute / Complete 语义
战斗输入链路治理为 BattleInputBuilder
```

---

### 9.6 第五阶段：工程治理期

目标：

```text
支撑 3-5 年持续演进
```

动作：

```text
稳定后再拆 asmdef
Runtime / Editor / ThirdParty 分离
建立模块负责人边界
CI 检查架构规则
禁止 UI 直接引用 Repository
禁止 Procedure 直接 SendProtobufData
禁止新代码 Resources.Load
```

## 10. 关键架构决策

### ADR-001：采用 Procedure FSM 作为统一流程骨架

决策：

```text
统一采用 Procedure FSM 管控启动、登录、Hub、场景、仿真、战斗流程。
```

原因：

```text
Procedure 能明确流程边界
可替代 GameMgr / SceneMgr partial 的流程堆叠
符合 GFX / UGF 项目经验
```

代价：

```text
旧项目需要 Adapter
团队必须遵守 Procedure 岗位职责
```

---

### ADR-002：引入 ApplicationService + DomainService 防腐层

决策：

```text
所有新业务操作必须通过 ApplicationService / DomainService。
```

原因：

```text
降低 UI 与业务耦合
控制 GameMgr / Repository / MrCSave 膨胀
支持渐进式治理
```

代价：

```text
初期代码层级增加
需要控制 Service 粒度，防止新 God Object
```

---

### ADR-003：DC / Repository 作为单一事实来源

决策：

```text
运行期数据由 DC 承载，实体数据由 Repository 承载。
```

原因：

```text
Dao / PlayerState / EntityRepository 都证明了集中状态管理的价值
但必须剥离 UI 操作和业务规则
```

代价：

```text
需要迁移旧写入路径
Presenter / DomainService 需要补齐
```

---

### ADR-004：统一资源入口，不立即强制替换底层

决策：

```text
新增 AssetService / ResourceFacade，兼容 Resources、UGF Resource、Addressables、AssetBundle。
```

原因：

```text
资源系统直接替换风险高
先收敛调用点，再逐步替换底层
```

代价：

```text
短期仍会存在多套底层资源实现
需要持续治理旧调用点
```

---

### ADR-005：运行态使用 Snapshot 隔离编辑态

决策：

```text
仿真运行、战斗运行不直接读取编辑态 Repository / PlayerState。
```

原因：

```text
避免运行态污染编辑态
降低保存、切换、回放、调试风险
```

代价：

```text
需要额外快照构建成本
快照同步边界要明确
```

## 13. 最终结论

玄盾流架构不是把旧项目推倒重来，而是在旧系统外建立一套清晰的新秩序。

它保留三类旧架构中的成熟能力：

```text
GFX / UGF 的 Procedure、Event、UI、Resource 能力
Dao / DC 的数据中心思想
UIWindowMgr 的 UI 栈管理经验
BattleField / Lua 的战斗扩展能力
DataSaver 的原子保存策略
EventProcessor + Handler 的事件扩展模型
```

同时规避旧架构的主要风险：

```text
GameMgr / MrCSave / Repository 膨胀
UI 直接写业务数据
Procedure 变成功能堆砌场
资源入口分散
运行态污染编辑态
事件滥用和生命周期泄漏
```

最终一句话：

> 玄盾流架构，是一套以流程定时、以领域守界、以数据归源、以快照隔离、以防腐续命的 Unity 大型项目渐进式架构。
